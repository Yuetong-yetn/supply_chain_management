# 供应链库存协同与智能补货管理系统项目文档

## 1. 系统概述

系统面向中小型连锁企业、校园商超、社区便利店等多仓库、多门店零售场景，以库存管理为核心，对商品从供应商采购进入仓库，再由仓库配送至门店的全过程进行统一建模、记录和分析。

项目通过需求分析和数据库建模，在商品、供应商、仓库、门店、采购订单、入库单、出库单、库存和库存流水等基础实体上，结合项目实际，补充了库存预警、门店补货申请、供应商管理与排行、统计分析、AI API 增强的智能补货建议、OceanBase 主数据库部署、系统状态检查、角色化演示界面和多端共享数据访问等内容。

## 2. 系统需求分析

### 2.1 业务背景

在连锁零售和校园商超等场景中，企业通常同时管理多个仓库和多个门店。商品从供应商采购后进入仓库，再由仓库根据门店需求进行配送。由于商品种类多、库存节点多、业务单据多，如果缺少统一的数据管理平台，容易出现以下问题：

- 仓库库存与门店库存信息不同步；
- 门店缺货后补货响应滞后；
- 仓库部分商品积压，占用仓储空间和资金；
- 出入库记录分散，库存变动难以追溯；
- 补货决策依赖人工经验，缺少数据分析支持；
- 多人、多电脑操作时，如果数据源不统一，容易出现记录不一致。

因此，本系统以关系数据库为核心，对供应链中的基础数据、业务单据、库存状态和分析数据进行集中管理，使系统能够支撑采购入库、门店补货、仓库出库、库存流水追踪和智能补货建议等业务。

### 2.2 用户角色分析

| 角色 | 主要需求 | 操作 |
|---|---|---|
| 系统管理员 | 维护基础数据，检查系统运行状态 | 维护基础数据、初始化数据、检查系统状态、查看全部模块 |
| 采购人员 | 管理供应商和采购订单，跟踪到货情况 | 查看供应商、维护供应商、创建采购入库单 |
| 仓库人员 | 处理入库、出库和库存调整 | 完成入库、执行出库、查询仓库库存 |
| 门店人员 | 查询门店库存并提交补货申请 | 创建补货申请、查看补货状态、跟踪发货和签收状态 |
| 业务主管 | 关注库存风险、供应商表现和经营分析 | 查看看板、库存预警、供应商排行、补货建议、统计分析 |

### 2.3 功能性需求

#### 2.3.1 用户登录、注册、角色化演示与系统状态检查

系统提供用户登录和新用户注册入口。用户可使用数据库中保存的账号密码登录，也可在前端注册新账号。注册时需要填写用户名、姓名、手机号、密码和角色；系统根据角色确定用户身份类型，仓库人员、门店人员会自动对应仓库或门店地点类型。

不同角色在前端看到的模块不同。系统管理员可查看全部模块；采购人员聚焦采购入库和供应商信息；仓库人员聚焦仓库库存、入库、出库和流水；门店人员聚焦本门店补货、签收和智能补货建议；业务主管负责审核补货申请、查看预警和统计分析。门店账号绑定具体门店后，前端仅展示该门店相关的库存、补货申请、出库签收和补货建议数据。

系统后端提供用户列表、创建、登录、修改和停用接口。注册功能复用用户创建接口，登录接口会校验数据库用户账号和密码，返回用户角色、地点类型、仓库 ID 和门店 ID 等基础信息。

系统状态模块用于查看后端服务、数据库连接和示例数据导入情况，便于使用前确认系统处于可运行状态。

#### 2.3.2 基础信息管理

系统需要对供应链基础数据进行统一维护，包括商品分类、商品信息、供应商信息、供应商与商品供货关系、仓库信息、门店信息和用户基础信息。当前前端提供商品、供应商、仓库和门店的新增入口；修改、删除和更完整的管理操作由后端接口提供。

#### 2.3.3 采购与入库管理

采购人员或仓库管理员能够查询采购订单和入库单。入库单完成后，系统需要校验入库单状态，根据入库明细增加对应仓库库存，将入库单状态更新为已完成，自动生成库存流水，并防止重复入库导致库存重复增加。

#### 2.3.4 门店补货与出库管理

门店人员可在前端提交补货申请。补货申请可由有权限角色审核通过或拒绝；审核通过后，可转换为仓库出库单。出库过程需要校验补货申请状态，生成出库单，发货前检查仓库可用库存，库存不足时拒绝出库，出库成功后扣减仓库库存，增加目标门店库存，并生成出库库存流水。门店签收后，出库单状态进入履约完成阶段。

#### 2.3.5 库存查询、预警与追溯

系统需要支持多仓库、多门店库存查询，并根据安全库存和最大库存生成预警。当前库存低于安全库存时提示缺货风险，当前库存高于最大库存或长期滞销时提示积压风险。所有库存变化通过库存流水记录，支持根据商品、单据或位置追踪库存流转过程。

#### 2.3.6 供应商管理与排行

项目中包含供应商基础信息、供应商商品供货关系、供应商采购排行和供应商评分快照等数据。系统可根据供货商品数量、供货周期、采购金额、延迟到货次数等字段展示供应商表现，辅助采购人员选择更合适的供应商。

#### 2.3.7 智能补货建议

系统根据当前库存、安全库存、近期销量和供应商信息生成补货建议。建议结果包括推荐补货数量、推荐供应商、缺货风险、风险等级、预计缺货天数和推荐理由。系统已接入 AI API，可在规则计算结果基础上生成更清晰的推荐说明，以提升系统的灵活性和泛化性；当外部 AI 服务临时不可用时，规则模型自动兜底，保证补货建议功能持续可用。

#### 2.3.8 统计分析与可视化

系统提供运营分析，把业务数据转化为管理视图：

- 商品数量、供应商数量、库存总量等关键指标；
- 商品库存排行；
- 门店需求热度；
- 供应商采购排行；
- 商品周转分析数据。

#### 2.3.9 多端共享访问

系统支持统一服务和统一数据库的访问方式。服务启动后，多台电脑访问同一套系统，一台电脑新增、修改或流转业务记录后，其他电脑刷新页面或重新查询即可看到相同数据。

### 2.4 非功能性需求

| 需求类型 | 说明 |
|---|---|
| 数据一致性 | 入库、出库、库存调整和库存流水需要保持一致 |
| 可追溯性 | 通过库存流水记录商品库存变动来源和目标 |
| 可维护性 | 系统按展示、接口、业务、数据模型和数据库访问职责分层 |
| 共享性 | 多台电脑访问统一服务时共享同一数据库记录 |
| 可扩展性 | 可扩展为跨仓调拨、分布式库存同步、高并发访问和预测模型优化等场景 |

## 4. 数据库逻辑设计

### 4.1 数据库设计原则

系统采用关系数据库进行建模，核心业务表满足第三范式（3NF），主要体现在：

- 基础数据独立建表，避免在业务单据中重复保存完整信息；
- 业务单据与明细分表，支持一张单据包含多个商品；
- 库存状态与库存流水分离，分别记录“当前状态”和“历史变动”；
- 统计分析类表保存的是从业务事实中派生出的统计结果，不作为核心交易表参与第三范式判断；
- 通过主键、外键、唯一约束和检查约束保证数据完整性。

### 4.3 关键表设计说明

#### 4.3.1 商品表

商品表保存商品基础信息，包括商品编码、名称、条码、类别、规格、单位、保质期、安全库存和启用状态。商品编码和条码设置唯一约束，保证商品主数据不重复。

#### 4.3.2 库存表

库存表保存商品在仓库或门店的当前库存状态。库存记录由商品、位置类型、仓库或门店共同确定。系统要求当前库存和冻结库存不得小于零，同一商品在同一库存节点仅保留一条当前库存记录。

#### 4.3.3 库存流水表

库存流水表用于记录所有库存变动，是系统可追溯性的核心。流水中保存变动类型、变动数量、变动前后数量、来源位置、目标位置和关联业务单据，使库存变化可以被追踪和复核。

#### 4.3.4 补货申请表

补货申请表记录门店补货需求，包括门店、商品、申请数量、申请原因、审核状态和关联出库单。补货申请通过审核后可转换为出库单，从而进入仓库发货流程。

#### 4.3.5 智能补货建议表

智能补货建议表记录系统生成的补货建议，包括门店、商品、当前库存、近期销量、平均日销量、安全库存、推荐补货数量、推荐供应商、风险等级、预计缺货天数、规则推荐理由、LLM 提供方、AI 调用状态和建议采用状态。该表用于保存分析结果，便于在运营驾驶舱中展示。

### 4.4 主键、外键与约束设计

主键：

- 各业务表均使用 `id` 作为主键。

唯一约束：

- 商品编码唯一；
- 商品条码唯一；
- 供应商名称唯一；
- 仓库编码唯一；
- 门店编码唯一；
- 采购订单号、入库单号、出库单号、补货申请号、库存流水号唯一。

外键：

- 商品关联类别；
- 库存关联商品、仓库或门店；
- 采购订单关联供应商和用户；
- 入库单关联供应商、仓库和用户；
- 出库单关联源仓库、目标门店和补货申请；
- 补货申请关联门店、商品和用户；
- 库存流水关联商品、仓库、门店、操作人和业务单据。

检查约束：

- 库存数量不得小于 0；
- 冻结库存不得小于 0；
- 供应商供货周期不得小于 0；
- 供应商质量评分限制在合理区间。

### 4.5 第三范式分析

系统核心业务表满足第三范式：

- 每张表描述单一业务实体；
- 非主属性依赖主键，而不依赖其他非主属性；
- 商品、供应商、仓库、门店等基础信息不重复存储在业务单据中，而是通过外键关联；
- 采购订单和采购明细、入库单和入库明细、出库单和出库明细分离，避免明细字段重复；
- 库存表只保存当前状态，库存流水表保存历史变动，职责分离。

分析类表如月度销售事实表和供应商评分快照表保存统计事实和评分快照，属于面向分析展示的派生数据表，不改变核心业务表满足第三范式的结论。

### 4.6 索引设计

除主键、唯一约束和外键形成的基础索引外，适合重点优化的查询字段包括：

- 商品编码、商品条码；
- 供应商名称；
- 仓库编码、门店编码；
- 采购订单号、入库单号、出库单号、补货申请号；
- 库存表中的商品与位置组合；
- 库存流水中的商品、交易类型和关联单据；
- 智能补货建议中的门店、商品和风险等级。


## 5. 功能设计

### 5.1 基础信息管理

基础信息管理为系统其他业务提供主数据支持。商品、供应商、仓库、门店和用户等数据被业务单据和库存记录引用，是采购、入库、出库和统计分析的基础。当前系统前端提供商品、供应商、仓库和门店的新增表单，新增后的数据会进入后续入库单、补货申请和统计展示；后端同时提供修改和删除接口，用于完整维护基础数据。

### 5.2 采购入库功能

采购入库功能用于处理供应商到仓库的商品流入。系统支持在前端新建入库单并完成入库操作。入库完成后，系统根据入库单明细增加仓库库存，并同步生成库存流水。该功能体现数据库事务一致性：库存增加、流水生成和单据状态更新必须共同成功。

### 5.3 门店补货功能

门店补货功能用于处理门店库存不足时的补货请求。系统支持在前端新建补货申请。门店提交申请后，系统记录申请商品、数量、原因和状态。申请审核通过后，可转换为出库单，由仓库执行发货。

### 5.4 仓库出库功能

仓库出库功能用于处理仓库向门店配送商品。系统在出库前检查仓库库存是否充足，库存不足时拒绝发货；发货成功后扣减仓库库存，增加门店库存，并生成出库流水。前端支持补货申请审核、转出库单、出库发货和签收等演示操作。

### 5.5 库存预警与追溯功能

库存预警功能根据当前库存、安全库存和最大库存判断风险。库存追溯功能通过库存流水记录入库、出库、调整等变动，使管理人员可以查询某个商品的库存变化来源。

### 5.6 供应商管理与排行

供应商相关能力主要体现为供应商基础数据管理、供应商商品供货关系、供应商采购排行和供应商评分快照。系统通过供货商品数量、平均供货周期和采购金额等信息展示供应商表现。

### 5.7 智能补货建议

智能补货模块根据库存、安全库存、近期销售和供应商数据生成补货建议。系统已接入 AI API，对规则计算得到的建议进行文本增强，输出推荐数量、风险等级、预计缺货天数、推荐理由和 AI 增强说明；当 AI API 调用失败时，系统保留规则模型结果并记录对应提供方状态。

### 5.8 统计分析与可视化

统计分析模块通过可视分析展示库存和经营情况，帮助管理者快速了解系统状态。前端展示商品库存排行、门店需求热度和供应商采购排行等信息；后端同时提供商品周转分析数据，用于扩展报表或后续前端图表展示。

## 7. 系统实现

### 7.1 用户注册、登录与角色化演示实现

前端提供登录与注册页面。登录时，系统调用后端用户登录接口校验数据库中的用户名和密码，并根据返回的角色进入对应工作台；注册时，系统调用用户创建接口保存新账号，密码在后端进行哈希处理。用户信息包含角色、地点类型、仓库 ID 和门店 ID，前端据此控制模块可见性和门店数据范围。门店人员登录后，页面聚焦本门店补货、签收和智能补货建议，避免与其他门店数据混在一起。

### 7.2 入库事务实现

入库完成时，系统在同一个业务事务中完成以下操作：

1. 查询入库单；
2. 校验入库单状态；
3. 查询入库明细；
4. 增加对应仓库库存；
5. 生成库存流水；
6. 更新入库单状态；
7. 提交事务。

如果过程中出现异常，则回滚事务，避免出现库存增加但流水未生成等不一致情况。

### 7.3 出库事务实现

出库发货时，系统完成以下操作：

1. 查询出库单；
2. 校验出库单状态；
3. 检查仓库库存是否充足；
4. 扣减仓库库存；
5. 增加门店库存；
6. 生成出库流水；
7. 更新出库单状态。

如果仓库库存不足，系统抛出业务异常，不允许库存被扣成负数。

### 7.4 补货申请实现

门店补货申请从创建到完成包括申请创建、审核通过或拒绝、转出库单、发货和门店签收等阶段。该流程把门店需求与仓库配送连接起来，使补货过程能够被记录、审核和追踪。转出库单时，系统支持指定仓库；如果未指定仓库，系统会在可用库存满足需求的仓库中选择剩余库存最充足的仓库，库存不足时拒绝生成出库单。

### 7.5 库存预警实现

库存预警根据当前库存、安全库存和最大库存判断。低于安全库存时生成缺货风险提示，高于最大库存或长期滞销时生成积压风险提示。预警结果用于运营驾驶舱展示，也为补货建议提供依据。

### 7.6 智能补货建议实现

智能补货建议模块先依据规则计算模型，调取库存和销售数据生成业务建议，再调用 AI API 对推荐理由进行增强。核心判断因素包括当前库存、安全库存、近 7 天销量、近 30 天销量、日均销量、预计缺货天数和供应商供货关系。建议结果保存推荐数量、风险等级、规则理由、AI 增强理由、LLM 提供方、是否成功调用 AI 和采用状态等字段，并在运营驾驶舱中展示。业务人员可通过接口将建议标记为已采用或已拒绝，便于后续分析建议质量。

### 7.7 规则计算模型、超参数与公式

系统中的预警事件和智能建议以规则计算模型为基础生成，AI API 主要用于对规则结果进行文本增强，不替代核心业务判断。主要超参数、阈值和公式如下。

#### 7.7.1 库存可用量与出库校验

库存记录包含当前库存、安全库存、冻结库存和最大库存。系统在出库、补货转出库和跨仓调拨时统一使用可用库存进行校验：

| 指标 | 公式或取值 | 说明 |
|---|---|---|
| 当前库存 | `current_quantity` | 当前地点账面库存数量 |
| 冻结库存 | `frozen_quantity` | 已被业务占用但尚未实际扣减的库存 |
| 可用库存 | `available_quantity = current_quantity - frozen_quantity` | 出库、调拨和自动选仓的库存判断依据 |
| 出库通过条件 | `available_quantity >= request_quantity` | 不满足时拒绝生成或执行出库 |
| 入库后库存 | `after_quantity = before_quantity + quantity` | 入库、签收等增加库存场景 |
| 出库后库存 | `after_quantity = before_quantity - quantity` | 发货、调拨等扣减库存场景 |

当系统自动选择补货出库仓库时，先筛选满足 `available_quantity >= request_quantity` 的仓库，再按 `available_quantity - request_quantity` 从大到小排序，选择发货后剩余库存最充足的仓库；若所有仓库均不满足条件，则拒绝生成出库单。

#### 7.7.2 库存预警规则

库存预警根据库存表中的 `current_quantity`、`safety_stock` 和 `max_stock` 计算。系统生成三类预警：

| 预警类型 | 触发公式 | 含义 |
|---|---|---|
| 严重缺货预警 `critical_stockout` | `current_quantity <= safety_stock * 0.5` | 当前库存不超过安全库存的一半 |
| 缺货预警 `stockout` | `safety_stock * 0.5 < current_quantity <= safety_stock` | 当前库存低于或等于安全库存 |
| 积压预警 `overstock` | `current_quantity >= max(max_stock, safety_stock * 4)` | 当前库存达到最大库存，或达到安全库存 4 倍以上 |

其中 `safety_stock` 可来自库存记录或商品默认安全库存；商品默认安全库存 `default_safety_stock` 的系统默认值为 10。`max_stock` 为库存地点的最大库存控制值；当该值低于安全库存 4 倍时，系统使用 `safety_stock * 4` 作为积压判断下限，避免最大库存配置过低导致误报。

#### 7.7.3 智能补货建议模型

智能补货建议以门店库存和最近一个月销售事实为输入。当前版本按最新销售月份作为最近 30 天窗口，并由 30 天销量估算 7 天销量和日均销量：

| 指标 | 公式或取值 | 说明 |
|---|---|---|
| 最近 30 天销量 | `recent_30_sales = latest_month_retail_sales` | 最新销售月份中该门店、该商品的零售销量 |
| 最近 7 天销量 | `recent_7_sales = recent_30_sales / 4` | 由最近 30 天销量近似换算 |
| 平均日销量 | `avg_daily_sales = recent_30_sales / 30` | 预测缺货天数和目标库存的基础 |
| 默认供应周期 | `lead_time_days = 3` | 未匹配供应商商品关系时采用 |
| 复核周期 | `review_period_days = 7` | 补货后到下一次复核的计划周期 |
| 缓冲天数 | `buffer_days = 3` | 防止需求波动的额外缓冲 |
| 目标覆盖天数 | `target_days = lead_time_days + review_period_days + buffer_days` | 补货后需要覆盖的销售天数 |
| 目标库存 | `target_stock = ceil(avg_daily_sales * target_days + safety_stock)` | 满足供应周期、复核周期、缓冲期和安全库存 |
| 建议补货量 | `recommended_quantity = max(0, target_stock - current_quantity)` | 当前库存已满足目标时建议量为 0 |
| 预计缺货天数 | `days_until_stockout = current_quantity / max(avg_daily_sales, 0.01)` | 日均销量为 0 时使用 0.01 防止除零 |

风险等级根据预计缺货天数和供应周期判断：

| 风险等级 | 判断条件 | 业务含义 |
|---|---|---|
| 高风险 `high` | `days_until_stockout <= lead_time_days` | 预计缺货时间不晚于供应到货时间，需要优先处理 |
| 中风险 `medium` | `lead_time_days < days_until_stockout <= lead_time_days + 3` | 预计缺货时间接近供应周期，需要关注 |
| 低风险 `low` | `days_until_stockout > lead_time_days + 3` | 短期缺货风险较低 |

系统将 `medium` 和 `high` 标记为 `shortage_risk = true`。补货建议的推荐供应商由供应商商品关系确定，排序优先级为：供应商最新评分从高到低、供货周期从短到长、供货价格从低到高。

#### 7.7.4 供应商评分模型

供应商评分根据供应商商品关系和采购、销售活动生成快照，分数范围固定为 0 到 100：

| 指标 | 公式或取值 | 说明 |
|---|---|---|
| 供应商品数 | `product_count = count(supplier_products)` | 该供应商关联的商品数量 |
| 平均供货周期 | `avg_lead_time_days = sum(lead_time_days) / product_count` | 无关联商品时为 0 |
| 平均准时率 | `avg_on_time_rate = sum(on_time_rate) / product_count` | 供应商商品关系默认值为 0.9 |
| 平均质量分 | `avg_quality_score = sum(quality_score) / product_count` | 供应商商品关系默认值为 8.0，约束范围为 0 到 10 |
| 采购金额 | `total_purchase_amount = sum(purchase_orders.total_amount)` | 用于供应商采购排行 |
| 延迟到货次数估算 | `delayed_arrival_count = max(0, int(product_count * (1 - avg_on_time_rate) * 10 ))` | 结合准时率估算 |
| 综合评分 | `score = 100 - avg_lead_time_days * 2 - delayed_arrival_count * 5 + avg_on_time_rate * 20 + avg_quality_score * 10` | 结果四舍五入到 2 位小数，并限制在 0 到 100 |

供应商排行按 `score` 从高到低排序；供应商采购排行按 `total_purchase_amount` 从高到低排序。评分来源字段根据数据情况记录为真实采购数据、销售估算或示例数据。

#### 7.7.5 调拨建议与库存再平衡规则

跨仓调拨建议以仓库库存为对象，按商品分组生成。系统先识别低库存仓和高库存仓：

| 类型 | 判断公式 | 说明 |
|---|---|---|
| 低库存仓 | `current_quantity < safety_stock` | 需要补充库存的仓库 |
| 高库存仓 | `current_quantity > max(max_stock, safety_stock * 3)` | 可作为调出来源的仓库 |
| 建议调拨量 | `min(high.current_quantity - high.safety_stock, low.safety_stock - low.current_quantity)` | 同时避免调出仓低于安全库存、调入仓超过安全库存需求 |

系统只保留 `suggested_quantity > 0` 的调拨建议。实际完成调拨时仍会再次校验源仓可用库存，防止并发或库存变更导致库存不足。

#### 7.7.6 统计分析与驾驶舱规则

统计分析模块对业务表进行聚合计算，形成驾驶舱指标：

| 指标 | 计算方式 | 说明 |
|---|---|---|
| 库存结构排行 | 按商品汇总 `sum(inventory.current_quantity)`，取前 20 名 | 展示库存数量最高的商品 |
| 缺货预警数 | 统计预警类型包含 `stockout` 的记录 | 包括普通缺货和严重缺货 |
| 积压预警数 | 统计 `warning_type = overstock` 的记录 | 展示库存积压风险 |
| 近期出库数量 | 汇总状态为 `shipped` 或 `signed` 的出库明细数量 | 反映近期已发货或已签收数量 |
| 高风险建议数 | 统计 `risk_level = high` 的补货建议 | 反映需优先处理的建议 |
| 商品周转表现 | 按商品计算 `avg(monthly_sales_facts.retail_sales)` 并降序排序 | 当前以平均月销售量表示周转表现 |
| 门店需求热度 | 按门店汇总 `sum(monthly_sales_facts.retail_sales)` 并降序排序 | 反映不同门店需求强弱 |
| 仓库流量趋势 | 按年月和仓库汇总 `sum(monthly_sales_facts.warehouse_sales)` | 用于展示仓库销售/流转趋势 |

驾驶舱摘要在 AI API 可用时调用模型生成摘要；当 AI API 不可用时，系统使用规则计算模型返回摘要，包含商品数、供应商数、缺货预警数、积压预警数和高风险补货建议数。

#### 7.7.7 AI 增强与系统级参数

AI 模块不改变上述规则公式，只对规则生成的补货理由或经营摘要进行文字增强。相关系统参数如下：

| 参数 | 默认值 | 说明 |
|---|---|---|
| `llm_provider` | `rule` | 默认使用规则模型；可切换 DeepSeek 或 Ollama |
| `ollama_model` | `qwen2.5:7b` | 本地 Ollama 默认模型 |
| `deepseek_model` | `deepseek-chat` | DeepSeek 默认模型 |
| `llm_timeout_seconds` | 30 | AI 请求超时时间 |
| `llm_max_retries` | 2 | AI 请求最大重试次数 |
| `cache_ttl_seconds` | 60 | 统计分析缓存时间 |
| `database_connect_timeout_seconds` | 3 | 数据库连接检查超时时间 |

生成补货建议时，只有前 5 条高风险建议会尝试进行 AI 文本增强；如果 AI API 配置缺失、网络异常或服务不可用，系统返回规则理由，并将提供方记录为 `rule`。

### 7.8 智能补货建议实现

智能补货建议模块先依据规则计算模型，调取库存和销售数据生成业务建议，再调用 AI API 对推荐理由进行增强。核心判断因素包括当前库存、安全库存、近 7 天销量、近 30 天销量、日均销量、预计缺货天数和供应商供货关系。建议结果保存推荐数量、风险等级、规则理由、AI 增强理由、LLM 提供方、是否成功调用 AI 和采用状态等字段，并在运营驾驶舱中展示。业务人员可通过接口将建议标记为已采用或已拒绝，便于后续分析建议质量。

### 7.9 OceanBase 主数据库与 AI API 接入实现

系统已完成 OceanBase 主数据库部署适配。后端通过 SQLAlchemy 和 PyMySQL 连接 OceanBase 的 MySQL 兼容协议，数据库连接、会话创建和运行状态检查集中在数据库访问层处理。系统健康检查接口会返回当前运行模式和脱敏后的连接信息，便于确认系统是否连接到主数据库。

系统已完成 AI API 接入。后端提供 LLM 路由层，可根据配置选择 DeepSeek、Ollama 或规则模型。补货建议和经营分析模块在生成业务结果后调用 AI 服务进行文本增强；如果 AI API 密钥、网络或外部服务不可用，系统自动使用规则结果返回，避免业务流程中断。

### 7.10 统计分析实现

统计分析模块将业务数据汇总为管理指标，包括商品数量、供应商数量、库存总量、库存排行、门店需求热度、供应商采购排行和商品周转分析数据。该模块面向经营管理者，帮助其快速判断库存风险和供应链运行情况。

### 7.11 多端共享实现

系统采用统一服务和统一数据库的方式支持多端共享。所有客户端写入同一份数据库，因此一台电脑新增商品、修改库存或创建补货申请后，其他电脑重新查询即可看到最新记录。

### 7.12 系统状态实现

系统状态页调用健康检查、数据库状态和示例数据状态相关接口，展示后端服务是否在线、数据库连接是否正常以及数据是否已导入。

### 8.3 风险与处理

- 如果依赖配置失败，按照 README 中的工程说明重新处理；
- 如果示例数据被改乱，重新执行数据库初始化和示例数据导入；
- 如果主数据库连接异常，检查 OceanBase 服务、数据库账号、网络端口和环境变量；必要时可使用本地应急数据库继续排查；
- 如果 AI API 调用失败，检查提供方、模型名称、API Key 和网络访问；规则模型会返回稳定结果；
- 如果多电脑访问失败，检查是否使用统一服务地址，以及防火墙是否放行服务端口。

## 11. 总结与不足

本项目完成了数据库课程所要求的主要内容：需求分析、概念模型设计、逻辑结构设计、约束设计、业务流程实现、示例数据导入、自动化测试和前端可视化展示。系统能够展示从供应商采购入库到仓库库存增加，再到门店补货、仓库出库、门店签收、门店库存增加和智能补货建议的完整业务闭环。

项目亮点包括：

- 数据库实体较完整，覆盖供应链核心业务；
- 入库、出库与库存流水结合，体现事务一致性；
- 库存预警和智能补货建议增强了决策支持能力；
- 统计分析页面能够直观展示经营数据；
- 演示登录和角色化导航能展示不同岗位视角；
- 支持统一服务和统一数据库下的多电脑共享访问；
- 提供完整示例数据、测试说明和演示脚本。

当前不足：

- 当前登录和角色控制定位为演示级能力，后端已有基础登录校验，完整令牌机制和复杂权限控制仍可继续完善；
- 前端已支持基础数据新增和主要业务流转演示，修改、删除和批量维护等完整后台管理能力仍主要依赖接口文档操作；
- AI 增强建议已接入，提示词效果评估和预测准确率验证还可以进一步加强；
- 多用户并发和 OceanBase 主数据库性能测试还可以进一步加强。

后续优化方向：

- 增加完整角色权限和审计日志；
- 增加一键部署能力；
- 增强补货预测模型和 AI 提示词评估；
- 增加报表导出功能；
- 增加移动端适配；
- 补充 OceanBase 主数据库环境下的索引优化和并发测试。
