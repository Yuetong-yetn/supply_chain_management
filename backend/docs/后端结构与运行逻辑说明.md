# 后端结构与运行逻辑说明

本文描述当前的供应链多智能体后端。它是单个 FastAPI 进程，不是微服务：Sisyphus 编排器在应用启动时注册领域 Agent、挂载路由并连接进程内事件订阅。

## 1. 技术栈与边界

- Web 框架：FastAPI
- 编排：`kernel/sisyphus/SisyphusOrchestrator`
- 数据访问：SQLAlchemy
- 数据库：OceanBase（MySQL 兼容）或 SQLite 回退
- 测试：当前尚未补齐正式自动化测试
- 启动：Uvicorn

正常请求路径为：

```text
FastAPI router -> Agent handler -> SQLAlchemy models/database
```

跨 Agent 协作必须通过 `kernel/common/event.py` 的事件总线，不能直接调用另一个 Agent 的处理器。

## 2. 目录职责

```text
backend/
├── main.py                 # FastAPI 入口、异常处理、Sisyphus 启动
├── kernel/
│   ├── common/             # 配置、数据库、认证、响应、事件与共享查询
│   └── sisyphus/           # 编排器和工作流
├── agents/                 # 12 个自包含领域 Agent
│   └── <agent>/
│       ├── agent.py        # Agent 元数据和注册接口
│       ├── events.py       # 事件名称与订阅
│       ├── router.py       # HTTP 参数和响应边界
│       ├── handler.py      # 业务规则与事务编排
│       └── models.py       # 此 Agent 拥有的 ORM 模型
├── schema/                 # SQLite 数据库与生成的 schema/seed 文件
└── docs/                   # 后端运维与结构说明

项目根目录的 `scripts/` 提供建库、示例数据生成、导入和演示重置命令；这些脚本不是后端运行时代码目录的一部分。

## 3. FastAPI 启动与 Agent 注册

在 `backend/` 目录启动：

```bash
uvicorn main:app --reload
```

`main.py` 会创建 FastAPI 应用和 `SisyphusOrchestrator`，安装统一异常处理器，并在启动事件中执行以下步骤：

1. 调用 `Base.metadata.create_all()` 确保已注册的模型表存在。
2. 调用 `agents.register_all_agents(orchestrator)` 创建 12 个领域 Agent。
3. 为每个 Agent 调用 `on_startup()`。
4. 将 Agent 路由和系统路由挂载到 FastAPI。

注册状态可通过 `GET /api/system/agents` 查询。该接口应返回 12 个 Agent：用户、商品、供应商、采购、库存、仓库、门店、履约、流水、分析、推荐和监控。

## 4. Agent 职责与 API

| Agent | 职责 | 主要 API / 事件 |
|---|---|---|
| `user_agent` | 登录、JWT、员工注册和身份查询 | `/api/users/*` |
| `product_agent` | 商品和品类 | `/api/products`、`/api/categories` |
| `supplier_agent` | 供应商和供货关系 | `/api/suppliers/*` |
| `procurement_agent` | 采购单和入库单 | 发布 `procurement.inbound.completed` |
| `inventory_agent` | 仓库与门店库存 | 订阅入库、发货和签收事件 |
| `warehouse_agent` | 仓库主数据 | `/api/warehouses` |
| `store_agent` | 门店主数据 | `/api/stores` |
| `fulfillment_agent` | 补货申请、出库、发货和签收 | 发布履约事件 |
| `transaction_agent` | 库存流水 | 订阅库存增减事件 |
| `analytics_agent` | 分析看板和统计 | `/api/analytics/*` |
| `recommendation_agent` | 规则补货建议 | `/api/recommendations/*` |
| `monitoring_agent` | 健康、数据库和 LLM 状态 | `/api/health`、`/api/health/db`、`/api/llm/status` |

## 5. 请求、事务与错误处理

Router 负责 Pydantic 请求解析、依赖注入和响应包装；业务规则在同一 Agent 的 `handler.py` 中执行。请求级 `get_db` 依赖在成功时提交事务，异常时回滚。

所有 API 响应使用以下格式：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

业务规则失败时抛出 `BusinessException`，由 `main.py` 转换为统一错误响应；验证错误由 FastAPI 的验证异常处理器返回 HTTP 422。

库存事件复用发起请求的 SQLAlchemy Session：采购入库会增加库存并发布库存增加事件，履约发货会扣减库存，门店签收会增加门店库存；`transaction_agent` 为库存变化写入流水。任一事件处理失败时，整个请求事务回滚。

## 6. 数据库与初始化

`kernel/common/config.py` 从 `backend/.env` 读取首选数据库、SQLite 回退地址、认证和 LLM 配置。`kernel/common/database.py` 在模块加载时选择可用引擎；`GET /api/health/db` 返回活跃方言和脱敏后的实际数据库 URL。

从项目根目录初始化课程数据：

```bash
python scripts/init_db.py --rebuild
python scripts/generate_example_data.py
python scripts/load_example_data.py
```

`--rebuild` 会删除并重建当前数据库，属于破坏性操作。SQLite 下生成的 `schema/schema.sql` 和 `schema/seed.sql` 是脚本产物，不应手工修改。

## 7. LLM 配置

`monitoring_agent` 的 `GET /api/llm/status` 返回当前配置的 provider、model 和可用性，不暴露 API 密钥。补货数量与风险仍由规则逻辑决定；当前推荐 Agent 保持规则驱动，不依赖外部模型调用。

## 8. 测试与检查

从 `backend/` 运行：

```bash
python -m py_compile main.py
python -c "import main; print(main.app.title)"
```

当前多智能体框架尚未补齐正式自动化测试，也不再保留临时接口自检测试。上线前至少需要覆盖应用启动、12 个 Agent 注册、健康检查、认证、采购入库、库存变动、履约出库、门店签收、库存流水、推荐和关键错误路径。

## 9. 开发约束

- 新增或改变 API 契约时，先更新根目录 `docs/api_contract.md`。
- 不要在 Agent 之间直接导入处理器；使用事件总线。
- Router 不实现库存扣减、补货审核或其他事务规则。
- 领域模型只由所属 Agent 管理；跨 Agent 读取应使用共享查询边界。
- 前端请求统一通过 `frontend/api.js`。
